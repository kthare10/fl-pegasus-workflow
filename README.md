# Federated Learning Pegasus Workflow

This directory contains a Pegasus WMS workflow for federated learning over
medical imaging-style data. It supports two modes:

- smoke mode: synthetic/sample data for quick DAG validation
- real-data mode: actual NIH ChestX-ray14 or TCIA-style image inputs with
  end-to-end image-model training

The workflow produces:

- `results/results.tar.gz`
- `results/paper_tables.csv`
- `results/provenance_summary.json`

## What This Workflow Does

The workflow is now hierarchical. The root workflow does setup and finalization,
and each federated learning round is emitted as its own Pegasus `SubWorkflow`.

The root-level structure is:

1. `prepare_manifest`
2. one preprocessing job per client
3. one round subworkflow per federated learning round
4. one final evaluation job
5. one `package_results` job

Intermediate models stay in scratch. Final metrics, manifests, and packaged
results are staged out.

Inside each round subworkflow, Pegasus runs:

1. one `train_client` job per client in parallel
2. one `aggregate` job that consumes all client updates for that round
3. one `bundle_round_outputs` job that exports the round checkpoint and round
   aggregation metric as a single child-workflow artifact

After each round subworkflow finishes, the parent workflow runs
`unpack_round_outputs` to restore:

- `models/round_{r}_global.pt`
- `metrics/round_{r}/global_round_{r}_aggregation.json`

This bundle/unpack handoff is deliberate. It gives Pegasus one child artifact
to propagate back to parent scratch, which is more reliable than exporting the
checkpoint and metric as separate child outputs.

In dual-branch mode, the workflow also runs:

1. one preprocessing branch for TCIA and one for NIH
2. one centralized baseline job per branch
3. one cross-dataset evaluation job after branch training finishes

## Repository Layout

```text
workflow_generator.py        # Generates workflow.yml and Pegasus catalogs
bin/                         # Python wrapper scripts used as Pegasus transformations
configs/                     # Smoke config and dataset-specific templates
Docker/                      # Base container recipe
data/                        # Small smoke-test input data
manifests/                   # Generated by prepare_manifest
preprocessed/                # Generated by preprocessing jobs

```

## Prerequisites

Run Pegasus generation and submission on the same host where you intend to run
`pegasus-plan`. This matters because `workflow_generator.py` writes absolute
paths into `sites.yml` and `transformations.yml`.

Required on that host:

- Python 3.10 or 3.11
- Pegasus WMS CLI and Python API
- HTCondor access for the `condorpool` site
- Apptainer/Singularity support if you use the default container mode
- A real container image available from the execution environment

Minimum Python packages:

```bash
pip install pegasus-wms PyYAML torch torchvision flwr psutil pynvml
```

If you want to run the smoke wrappers locally outside Pegasus:

```bash
pip install numpy pandas scikit-learn matplotlib seaborn Pillow pydicom SimpleITK torch torchvision flwr psutil pynvml
```

## Container Image Requirement

If `container_image` is set, Pegasus will run the workflow inside that
Singularity/Apptainer image. The default template uses a placeholder image:

```json
"container_image": "docker://username/fl-medical-imaging:latest"
```

You must replace this with a real image before running on Pegasus. Available
recipes are:

- CPU image:
  [Docker/FLMedicalImaging_Dockerfile](/Users/kthare10/codex/escience-fl/fl-pegasus-workflow/Docker/FLMedicalImaging_Dockerfile:1)
- GPU image:
  [Docker/FLMedicalImaging_GPU.Dockerfile](/Users/kthare10/codex/escience-fl/fl-pegasus-workflow/Docker/FLMedicalImaging_GPU.Dockerfile:1)

Example:

```bash
cd fl-pegasus-workflow
docker build -t <your-user>/fl-medical-imaging:latest -f Docker/FLMedicalImaging_Dockerfile .
docker push <your-user>/fl-medical-imaging:latest
```

GPU example:

```bash
docker build -t <your-user>/fl-medical-imaging:gpu -f Docker/FLMedicalImaging_GPU.Dockerfile .
docker push <your-user>/fl-medical-imaging:gpu
```

Then update `configs/experiment.yaml`:

```json
"container_image": "docker://<your-user>/fl-medical-imaging:latest"
```

If your execution site already has Python plus the required packages installed,
you can disable container usage by setting:

```json
"container_image": ""
```

For GPU execution, set:

```json
"container_image": "docker://<your-user>/fl-medical-imaging:gpu",
"request_gpus": 1
```

`workflow_generator.py` already propagates `request_gpus` to `train_client`
and `train_centralized` jobs so HTCondor can match them onto GPU-capable slots.

## Real Dataset Support

The workflow now trains on resized image tensors:

- NIH preprocessing loads real chest X-ray images and emits normalized image
  records
- TCIA preprocessing loads real DICOM/volume/image files and emits normalized
  image records
- training uses PyTorch image backbones with Flower-compatible aggregation
- augmentation, backbone freezing, optimizer/scheduler selection, class-weighted
  loss, gradient clipping, and resource monitoring are configurable

## FL Runtime

Pegasus still handles orchestration, while the learning stack now uses:

- PyTorch for model initialization, local training, and evaluation
- torchvision `resnet18` as the default end-to-end image backbone
- Flower-compatible parameter aggregation in `bin/aggregate.py`
- optional FedProx local training via `aggregation: "fedprox"` and
  `fedprox_mu`
- centralized pooled-data baseline training via `bin/train_centralized.py`
- periodic CPU/GPU/RAM monitoring in training jobs

Supported `model_name` values:

- `resnet18`
- `torch-mlp`
- `feature-mlp`
- `linear-smoke`

`resnet18` is the intended default for end-to-end image training. The MLP
variants remain available for compatibility and smoke tests.

Additional runtime controls now supported in configs include:

- `optimizer`: `sgd`, `adam`, or `adamw`
- `scheduler`: `none`, `step`, or `cosine`
- `gradient_clip_norm`
- `pretrained`
- `freeze_backbone`
- `unfreeze_backbone_epoch`
- `class_weighted_loss`
- `monitor_interval_seconds`

## Experiment Config Pack

Experiment templates live under
[configs/experiments](/Users/kthare10/codex/escience-fl/fl-pegasus-workflow/configs/experiments:1).

Important note: some older experiment JSON files were authored before the
current image-based `resnet18` runtime and `gpu-flwr` container became the
default validation path. In practice, the safest way to launch an experiment
is:

1. start from a known-good current config that already runs on the active
   Pegasus/HTCondor environment
2. carry over the experiment-specific values such as `rounds`,
   `local_epochs`, `num_clients`, and `aggregation`
3. keep runtime fields aligned with the current stack:
   - `container_image: docker://...:gpu-flwr`
   - `model_name: resnet18`
   - `request_gpus: 1`
   - optimizer/scheduler/class-weighted-loss settings

This matters because historical experiment configs may still reference older
container tags or feature-based smoke models such as `linear-smoke`.

### NIH ChestX-ray14

Use [configs/nih_cxr14.yaml](/Users/kthare10/codex/escience-fl/fl-pegasus-workflow/configs/nih_cxr14.yaml:1) and set:

- `data_root`
  Root directory containing the real image files
- `metadata_files`
  Path to `Data_Entry_2017_v2020.csv`
- `container_image`
  A real published image built from the Dockerfile

The manifest builder will:

- read `Image Index`, `Patient ID`, and `Finding Labels`
- recursively resolve image files under `data_root`
- assign clients deterministically by patient
- preserve patient-level train/val/test splits

Example:

```bash
cd fl-pegasus-workflow
cp configs/nih_cxr14.yaml configs/nih_cxr14.local.yaml
```

Edit `configs/nih_cxr14.local.yaml`:

```json
{
  "dataset_name": "nih-cxr14",
  "data_root": "/data/ChestXray14/images",
  "metadata_files": ["/data/ChestXray14/Data_Entry_2017_v2020.csv"],
  "container_image": "docker://<your-user>/fl-medical-imaging:latest",
  "num_clients": 5,
  "rounds": 5
}
```

Then run:

```bash
python3 workflow_generator.py \
  --config configs/nih_cxr14.local.yaml \
  --output workflow.yml
pegasus-plan --submit -s condorpool -o local workflow.yml
```

### Important Real-Data Behavior

- If `allow_synthetic_fallback` is `false`, the workflow will fail instead of
  silently switching to fake data.
- Real NIH and TCIA configs now set `allow_synthetic_fallback` to `false`.
- For real datasets, missing metadata or bad image paths should be treated as
  configuration errors, not as smoke-test conditions.

## Generate the Pegasus Workflow

Generate the workflow on the host where you will run Pegasus:

```bash
cd fl-pegasus-workflow
python3 workflow_generator.py \
  --config configs/experiment.yaml \
  --output workflow.yml
```

This writes:

- `workflow.yml`
- `replicas.yml`
- `transformations.yml`
- `sites.yml`
- `pegasus.properties`

If `python3 workflow_generator.py` fails with
`ModuleNotFoundError: No module named 'Pegasus'`, install `pegasus-wms` into
that Python environment first.

## Review the Generated Catalogs

Before submission, check these generated files:

- `sites.yml`
  Confirms the `local` site scratch/storage directories and the `condorpool`
  execution site.
- `transformations.yml`
  Confirms each wrapper script PFN points to the correct absolute path on the
  submission host.
- `replicas.yml`
  Confirms the config file and metadata inputs are registered.
- `workflow.yml`
  Confirms the DAG was regenerated from the current source.

If you generate on one machine and submit on another, those absolute PFNs will
be wrong. Regenerate on the submission host instead of copying old catalogs.

## Submit with Pegasus

From `fl-pegasus-workflow/`:

```bash
pegasus-plan --submit -s condorpool -o local workflow.yml
```

Pegasus prints a run directory. Save it. You will need it for monitoring and
debugging.

Typical follow-up commands:

```bash
pegasus-status <run-dir>
pegasus-analyzer <run-dir>
pegasus-remove <run-dir>
```

## How To Know It Worked

When the workflow completes successfully, the staged outputs should appear in
the local storage area configured in `sites.yml`. In this repository layout
that is:

```text
fl-pegasus-workflow/output/
```

You should find:

- `output/manifests/dataset_manifest.csv`
- `output/manifests/client_manifest.csv`
- `output/manifests/splits.json`
- `output/models/model_config.json`
- `output/metrics/final_evaluation.json`
- `output/results/paper_tables.csv`
- `output/results/provenance_summary.json`
- `output/results/results.tar.gz`

If you are also running the manual smoke path, separate local wrapper outputs
from Pegasus staged outputs. `run_manual.sh` writes directly under
`manifests/`, `metrics/`, `models/`, and `results/`, while Pegasus stages final
artifacts under `output/`.

## How Many FL Rounds?

The number of federated learning rounds is controlled by the `rounds` field in
the config file.

Current defaults:

- smoke config: `configs/experiment.yaml` uses `2` rounds
- NIH template: `configs/nih_cxr14.yaml` uses `5` rounds
- TCIA template: `configs/tcia_lidc.yaml` uses `3` rounds

Every round does:

1. one local training job per client
2. one aggregation job

So if you use `5` clients and `5` rounds, you will get:

- `25` client training jobs
- `5` aggregation jobs
- plus manifest, preprocessing, initialization, evaluation, and packaging jobs

## Common Failure Modes

### `ModuleNotFoundError: No module named 'Pegasus'`

Cause:
The Python environment used to run `workflow_generator.py` does not have
`pegasus-wms` installed.

Fix:

```bash
pip install pegasus-wms PyYAML
```

### Planner failure due to basename conflict in CondorIO

Cause:
Two inputs with the same basename are staged into one job. Pegasus CondorIO does
not allow that, even if the files live in different logical directories.

This workflow specifically avoids that by naming round metrics like:

- `metrics/round_001/round_001_aggregation.json`
- `metrics/round_002/round_002_aggregation.json`

If you add new staged files to a single job, keep their basenames unique.

### Placeholder container image cannot be pulled

Cause:
`configs/experiment.yaml` still points at
`docker://username/fl-medical-imaging:latest`.

Fix:
Build or publish a real image and update the config before regenerating the
workflow.

### Workflow generated on the wrong host

Cause:
`transformations.yml` and `sites.yml` contain absolute paths from another
machine.

Fix:
Regenerate `workflow.yml`, `sites.yml`, `transformations.yml`, `replicas.yml`,
and `pegasus.properties` on the Pegasus submission host.

## Recommended Run Sequence

Use this order every time:

1. Update `configs/experiment.yaml` with a real `container_image`.
2. Install Pegasus Python dependencies on the submission host.
3. Run `bash ./run_manual.sh` once to validate the smoke workflow locally.
4. Regenerate the Pegasus catalogs with `python3 workflow_generator.py`.
5. Inspect `sites.yml` and `transformations.yml` for correct host paths.
6. Submit with `pegasus-plan --submit -s condorpool -o local workflow.yml`.
7. Monitor with `pegasus-status <run-dir>`.
8. If the run fails, inspect it with `pegasus-analyzer <run-dir>`.
9. Collect final artifacts from `output/results/`.
